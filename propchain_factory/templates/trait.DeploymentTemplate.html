Expand description

Common behavior every deployment template must provide.

Keeping this as a trait (rather than just encode_params per struct) means callers can validate-then-encode through one interface instead of hand-rolling the same two calls for every contract type.

fn encode_params(&self) -> Vec<u8> ⓘ

SCALE-encode the constructor parameters, in the exact order the target contract’s constructor expects them.

Provided Methods§

Source

fn validate(&self) -> Result<(), TemplateError>

Sanity-check the parameters before they’re ever encoded/deployed. Default impl is a no-op so templates with no extra constraints don’t need to implement this.

Source

fn try_encode_params(&self) -> Result<Vec<u8>, TemplateError>

Convenience: validate, then encode. This is what most callers should actually use.
